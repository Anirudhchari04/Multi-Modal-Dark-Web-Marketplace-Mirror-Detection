1. Executive Summary: The Paradigm Shift

We have successfully transitioned the system from a basic "Prototype Comparator" to a Production-Grade Repository Scanner.

  • Old Way (Phase 1): You had to manually provide a "Trusted URL" and a "Suspect URL" every single time. It was a 1-to-1 comparison that required you to already know the real site.
  • New Way (Phase 2): You build a Local Intelligence Repository. You ingest trusted sites once. Then, you can scan any suspicious link against your entire database. The system calculates similarity scores against every known market to find the true identity.

2. System Architecture

A. Network Layer (The Tor Tunnel)

The system does not use the standard internet. It routes all traffic through a SOCKS5 Proxy tunnel check. * Auto-Detection: verify_tor.py logic automatically scans local ports. * Checks Port 9050 (System Tor Service). * Checks Port 9150 (Tor Browser Bundle). * Routing: All .onion requests in pillar_1_http.py and crawler.py are legally routed through this tunnel, anonymizing the analysis.

B. The Ingestion Engine (crawler.py)

This is the "Data Entry" pipeline that builds your database. 1. Request: Connects to the target .onion via Tor. 2. Fingerprint: * HTTP: Measures response time stability (mean, std dev) and server headers. * HTML: Captures the raw DOM structure (placeholder for deep DOM tree analysis). * PGP (Smart Hunt): * Attempt 1: Regex scan the Homepage for -----BEGIN PGP.... * Attempt 2: If failed, automatically brute-force common locations: /pgp.txt, /key.txt, /security.txt. 3. Normalization: Converts raw data into a SiteSignature object. 4. Persistence: Saves the json blob to fingerprints.json via storage.py.

C. The Comparison Engine (Scanner)

When you scan a suspect URL, we don't just "diff" the text. We use a Feature Similarity Vector.

Pillar 1: HTTP Fingerprinting (Weight: 30%)

We compare the behavior of the server. * Status Code (40%): Does it return 200 OK like the real site, or a 403/404? * Headers (30%): Does it use the same Server: nginx or X-Powered-By headers? * Timing (30%): We calculate the "Response Time Ratio". If the suspect site is 5x slower or faster than the known site, it's penalized. Features are normalized to handle darknet latency fluctuations.

Pillar 4: PGP Cryptography (Weight: 70%)

This is the "Golden Signal". We use pgpy to perform deep inspection, not just string matching. * Primary Fingerprint: Do the Master Key IDs match? (100% Score if yes). * Subkey Overlap: Do the signing/encryption subkeys match? (95% Score). * Key Attributes: If keys are missing, we check if the Algorithm (RSA-4096) and Creation Date align. A phishing site generating a fresh key today will likely fail the "Creation Date" check against a market founded in 2022.

Pillar 7: Deep Code Analysis & Heuristics

The system performs forensic analysis on the site's underlying architecture. * Code Signature: Generates a deterministic cryptographic hash of the site's unique architectural footprint, allowing for instant identification of codebase forks. * Structure Integrity: A specialized heuristic engine that scores the site's DOM and script density against known "clean" marketplace templates to detect anomalies. * Backend Latency: Precision measurement of the hidden service's processing time, utilizing statistical noise filtering to isolate server speed from Tor network lag. * Search Query Time (P99): Estimates the database load capability by measuring 99th percentile response times, distinguishing high-performance real markets from low-resource phishing clones.

3. Data Structures (signatures.py)

The system understands the world through Signatures:

class SiteSignature:
    url: str                # The .onion address
    label: str              # e.g., "AlphaBay"
    http: HTTPSignature     # {response_time, headers, status_code}
    pgp: PGPSignature       # {fingerprints[], subkeys[], algorithms[]}
    html: HTMLSignature     # {dom_depth, tags}

This decoupled structure allows us to add new pillars (like JA3 TLS fingerprinting) without breaking the database.

5. Troubleshooting & FAQ

Q: My Scan Score is 0%, but it's the same site! * Reason: Did you capture the PGP key? * Fix: Check if the site hides its key. Try adding /pgp.txt to the URL manually during Ingest if the Smart Hunt failed.

Q: "Connection Failed" error? * Reason: Your Tor Browser might be closed. * Fix: Open Tor Browser. Ensure it's connected. Retry.

Q: PGP Extraction Failed log warning? * Status: Normal. * Impact: Non-fatal. The system will rely on HTTP/HTML pillars instead.
